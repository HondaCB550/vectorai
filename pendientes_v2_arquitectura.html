arquitectura replanteada Junio 2026 — Nueva estrategia
🔄 Cambio de paradigma:
De "comparar proveedores entre sí" → "Matchear cada proveedor CONTRA tu base de datos de materiales"
Tu estructura (código + detalle + unidad) es la fuente de verdad. Las unidades se validan automáticamente.
Arquitectura — Nuevo flujo de datos
A1
Upload de Base de Datos de Materiales
Usuario sube su archivo Base de Presupuestos → hoja Materiales. Sistema extrae: CÓDIGO, ITEM, DETALLE, UNIDAD, MARCA. Guarda en tabla `base_materiales` por user_id.
arquitectura frontend database
A2
Matcheo contra Base Propia (no inter-proveedor)
Para cada PDF de proveedor, el sistema matchea cada ítem CONTRA la base de materiales del usuario. Score = similitud descripción. No se comparan proveedores entre sí.
arquitectura backend
A3
Validación de Unidades
Cada match calcula: unidad_esperada (de base_materiales) vs unidad_cotizada (del PDF). Si NO coinciden → FLAG AMARILLA en el resultado. Ej: base="bolsa", PDF="m³" → ⚠️ Revisar.
arquitectura backend
A4
Descarga con Columnas de Revisión
Excel descargable con columnas: COD_BASE, DESCRIPCIÓN_BASE, UNIDAD_BASE | COD_PDF, DESC_PDF, UNIDAD_PDF, PRECIO_SIN_IVA | SCORE, FLAG_UNIDAD, DECISIÓN, CÓD_CORRECTO. Usuario revisa y corrige en Excel.
arquitectura frontend
A5
Re-upload de Decisiones Revisadas
Usuario descarga → revisa → corrige DECISIÓN + CÓD_CORRECTO en Excel → sube el mismo archivo. Sistema importa equivalencias confirmadas a tabla `equivalencias` por proveedor + user_id.
arquitectura backend
A6
Aprendizaje Incremental
Cada vez que un usuario confirma una equivalencia (DECIDIÓN=CARGAR), se guarda en tabla `equivalencias`. Próximas análisis de ese proveedor (mismo usuario) benefician del scoring anterior. Mismo proveedor, otros usuarios → opcional (plan Advance).
arquitectura backend database

Nuevo trabajo — Implementación v2
N1
Tabla `base_materiales` en Supabase
Campos: id, user_id, codigo, item, detalle, unidad, marca, created_at. RLS: usuario solo ve los suyos. Índice en (user_id, codigo).
nuevo database
N2
Endpoint POST /upload-base-materiales
Recibe archivo .xlsx o .json con materiales del usuario. Extrae hoja "Materiales", parsea, valida estructura (código, unidad obligatorio), guarda en base_materiales. Devuelve count y warnings.
nuevo backend
N3
Endpoint POST /analizar-v2
Recibe PDFs. Para cada PDF: extrae, matchea CONTRA base_materiales del usuario (no contra master global). Devuelve: matches con FLAG_UNIDAD si unidades no coinciden, sin_match, comparativa. Guarda en tabla `comparativas`.
nuevo backend
N4
Exportar a Excel con columnas de revisión
En lugar de tabla comparativa tradicional: formato BORRADOR (COD_BASE | COD_PDF + DESC_PDF + UNIDAD_PDF + PRECIO | SCORE + FLAG_UNIDAD | DECISIÓN + CÓD_CORRECTO). Usuario edita en Excel.
nuevo backend ui
N5
Endpoint POST /upload-decisiones
Recibe Excel revisado. Parsea columnas DECISIÓN + CÓD_CORRECTO. Para cada CARGAR/CAMBIAR: inserta en tabla `equivalencias` (cod_prov, cod_int, proveedor, user_id). Devuelve confirmación.
nuevo backend
N6
Frontend: Página de upload de base
Interfaz para subir Base de Presupuestos. Muestra preview: cantidad de ítems, rubros, unidades detectadas. Opción re-upload (reemplaza base anterior).
nuevo frontend
N7
Frontend: Visualizar flags de unidades
En la comparativa: si FLAG_UNIDAD=true, mostrar banner naranja "⚠️ Unidad mismatch". Ejemplificar: "Base: bolsa | PDF: m³". Permitir descargar y corregir en Excel antes de confirmar.
nuevo frontend
N8
Frontend: Workflow de revisión y re-upload
Botón "Descargar para revisar" → Excel → usuario edita → "Subir decisiones" → sistema importa equivalencias. Confirmación visual: "X nuevas equivalencias guardadas".
nuevo frontend

📋 Notas de implementación

1. Reusable vs. Nuevo: R1–R5 no se tocan. Se integran directamente en v2. N1–N8 es trabajo nuevo.

2. Base de Materiales: Ojo con duplicados. Si usuario sube base dos veces, ¿reemplazamos o mergeamos? Decisión: reemplazar (usuario controla versiones en su Excel).

3. Unidades: Comparar string normalizado (uppercase, trim). Si base="BOLSA" y PDF="BOLSAS" → OK. Si base="BOLSA" y PDF="M3" → FLAG.

4. Equivalencias por usuario + proveedor: Tabla `equivalencias` con UNIQUE (cod_prov, proveedor, user_id). Un usuario puede tener distinto mapeo que otro para el mismo proveedor.

5. Migración de datos: Borradores actuales (Excel) + decisiones_usuario.json → se pueden importar con seed_supabase_equivalencias.py como punto de partida.
