Usuario sube PDFs. Sistema matchea automáticamente contra BD interna usando códigos de proveedores (95% confianza). Lo que no matchea, nosotros lo validamos 1x/día. Flywheel: más datos propios → mejor matching → usuarios satisfechos.
Visión General
El Cambio Fundamental
VectorAI no es un "comparador entre proveedores". Es un sistema de matching inteligente que reconoce automáticamente qué material está cotizando cada proveedor, usando:

✓ Códigos internos de proveedores (95% confianza)
✓ Denominaciones con alias (70-90% confianza)
✓ IA para agrupar pendientes (60-85% confianza)
✓ Human review para validar nuevos materiales
Bootstrapping: 1-2 meses
No esperamos perfección día 1. El sistema mejora cada día porque:

• Usuarios suben PDFs → datos para entrenar
• Nosotros pedimos presupuestos a otros corralones → más datos
• Descubrimos códigos nuevos, denominaciones, patrones
• Validamos y agregamos a la BD
• Próximos usuarios: mejor matching

codigos_proveedores
La tabla "mágica" de matching automático. Mapea códigos internos de proveedores a nuestros códigos de materiales.

Estructura:
Campo Tipo Ejemplo
proveedor string Baukraft
codigo_proveedor string BK-4521
codigo_material string FK 001-A
marca_asignada string Marca A
confianza int 95 (validado)
validado_por email team@vectorai.com
created_at timestamp 2026-06-21
UNIQUE (proveedor, codigo_proveedor)
ℹ️ Clave compuesta: Previene que el mismo proveedor tenga dos códigos para el mismo material. Pero SÍ permite que dos proveedores usen el mismo código para materiales distintos (porque cada uno es independiente).
material_denominaciones
Alias y variaciones de nombres. Permite matchear "Codo PVC", "Codo 45-40", "Codo PVC A90" contra el mismo material.

Estructura:

origen string usuario_baukraft | pdf_bonora
confianza int (0-100) 98 (encontrado 50 veces)
frecuencia_encontrada int 50
created_at timestamp 2026-06-21

Matching Multi-Nivel
Nivel 1: Código de Proveedor (95%)
Sistema busca en tabla codigos_proveedores

✓ ENCONTRADO
→ Match automático, confianza 95%, no requiere revisión
→ Usuario ve checkmark, se confirma automáticamente

✗ NO ENCONTRADO
→ Pasa a Nivel 2
Nivel 2: Denominación Fuzzy (70-90%)
Sistema busca en tabla material_denominaciones

✓ ENCONTRADO (score > 85)
→ Match probable, confianza 87%, usuario revisa

⚠️ DUDOSO (score 60-85)
→ Usuario elige entre opciones

✗ NO ENCONTRADO (score < 60)
→ Pasa a Nivel 3
Nivel 3: Agrupamiento IA (60-85%)
IA busca denominaciones similares en materiales_pendientes de hoy (otros PDFs)

💡 Ejemplo:
Baukraft: "Tornillo #8x1 inox"
Bonora: "Tornillo #8x1"""
Carosio: "Tornillo acero #8"

IA: "Estos 3 probablemente sean lo mismo (91% confianza)"

Usuario ve: [✓ Agrupar] [✗ Son distintos]
Nivel 4: Sin Match (0%)
No hay código, no hay denominación, IA no encuentra similares.

→ Item se agrega a materiales_pendientes
→ Usuario ve: "No está en base, será revisado por nosotros"
→ Mañana nosotros decidimos qué hacer

¿Por qué funciona?
El matching depende de DATOS. Cada PDF de usuario o presupuesto que cargamos es una oportunidad de descubrir:

• Nuevos códigos de proveedores
• Nuevas denominaciones
• Patrones de descripción
• Nuevos materiales

Cuantos más datos, mejor entrena el sistema. No es magia, es acumulación inteligente.
Tablas en Supabase (Resumen)
5 Tablas Principales
1. materiales_validados
La fuente de verdad. 915+ materiales. Read-mostly. Actualizada por nosotros.

2. material_denominaciones
Alias y variaciones de nombres. Crece continuamente. Mejora fuzzy match.

3. codigos_proveedores
La tabla "mágica". Mapeo proveedor→código interno→material nuestro. UNIQUE (proveedor, codigo_proveedor).

4. materiales_pendientes
Nuevos encontrados. Nosotros los revisamos y movemos a validados.

5. precios_historicos
Data warehouse. Cada precio que ve el sistema se guarda aquí. Base para futuros análisis.

Mapeo proveedor + descripción → material validado + marca. Para usuarios (si aplica).

comparativas

Trigger: cleanup_old_comparativas

Riesgos y Consideraciones
¿Colisión de Códigos?
Riesgo: ¿Pueden dos proveedores usar el MISMO código para DIFERENTES materiales?

Respuesta: SÍ es posible, pero es raro.

Ejemplo (hipotético):
• Baukraft BK-100 = Acero estructural 100mm
• Bonora BON-100 = Pintura latex blanca 10L

Solución: Clave compuesta UNIQUE (proveedor, codigo_proveedor)
Esto permite que cada proveedor tenga BK-100 y BON-100 por separado.

Double-check:
Si el usuario escribe "BK-10" (incompleto), sistema:
1. Busca en codigos_proveedores
2. Si encuentra múltiples (BK-100, BK-101, etc.)
3. Usa denominación como desempate
4. "BK-10, Acero" → matchea con "Acero estructural"
False Positives
Riesgo: IA agrupa materiales que NO son lo mismo.

Mitigación:
• Usar umbral de confianza alto (>85% para auto-agrupar)
• Usuario revisa agrupamientos antes de confirmar
• Nosotros validamos en Human Loop
• Datos incorrrectos nunca entran a materiales_validados sin revisión
Escalabilidad de Denominaciones
Riesgo: Tabla material_denominaciones crece muy rápido.

Mitigación:
• Crear índice en (codigo_material, denominacion)
• Limpiar duplicados periódicamente
• Agradecer solo aliases con confianza >70%
• Monitorear tamaño de tabla
Bootstrapping Lento
Riesgo: ¿Y si no hay suficientes usuarios/datos en primeras semanas?

Mitigación:
• Nosotros NO esperamos. Cargamos datos propios.
• Solicitamos presupuestos a otros corralones.
• Procesamos nosotros mismos los datos.
• El sistema mejora incluso sin usuarios iniciales.
Validación de Códigos Duplicados
Riesgo: ¿Qué pasa si un usuario sube el MISMO PDF dos veces?

Mitigación:
• Sistema detecta hash del PDF
• No procesa dos veces
• O permite re-procesar si el usuario desea
